1 · Auto-Tolerance — Statistics-Driven Row Tolerance

Why This Was Needed

Until this session the engineer had to guess the row tolerance (the per-row drift the engine accepts before classifying a column into a different row). 60 cm worked on some plans, 130 cm on others, and the wrong value caused either over-merging (two visual rows fused into one) or chain-bridging (one row chaining across the entire plan). The user explicitly demanded a generic solution that works office-to-office, project-to-project, without manual tuning.

1.1 — Algorithm

The detector treats the per-axis distribution of column-to-column gaps as bimodal:

  • Mode A — In-row drift (small): residual modelling error along the row direction. Expected 5–80 cm.
  • Mode B — Between-row separation (large): true structural row spacing. Expected 3–10 m.

For each axis (X and Y), per zone, the algorithm:

  1. Sorts groups by the axis value, computes the consecutive gap series.
  2. Sorts the gaps ascending, scans for the largest relative jump (ratio between consecutive sorted gaps).
  3. Validates the jump using absolute thresholds — both sides must respect minimum drift / minimum separation.
  4. If a clean break is found → drift = max gap below break, separation = min gap above break, confidence = high.
  5. If no break but smallest gap is already large → "pristine grid"; suggest the floor (~30 cm).
  6. Otherwise fall back to a 95th-percentile heuristic with low confidence.

1.2 — Final Suggestion Formula

// Conservative headroom on top of the worst observed in-row drift,
// capped at the midpoint between drift and the next row.
double tight    = drift * 1.3;
double midpoint = (drift + separation) / 2.0;
double feet     = Math.Min(tight, midpoint);

// Industry clamps
return Math.Clamp(feet, 30 cm, 200 cm);

The 1.3 multiplier gives a safety margin without inflating the tolerance to the midpoint when the next row is far away. The midpoint is only used as a hard ceiling for very tight grids. The clamps reflect office-grade modelling practice (≥30 cm because no model is perfectly clean; ≤200 cm because true row spacing rarely drops below 4 m and we always want a clear gap).

1.3 — Cross-Axis Stability

Real plans often have very regular X-spacing but irregular Y-spacing (corridors, stairs). If the detector were to inspect only the axis perpendicular to the chosen sort direction, the same plan would propose different tolerances depending on whether the user chose "Top → Bottom" or "Bottom Left → Right". This was experienced in testing.

Fix: the detector analyses both axes and picks the most demanding confident estimate. This is justified by the engineering insight that modelling drift is broadly isotropic — a draftsman who misaligns columns by 30 cm in X is just as likely to misalign them by 30 cm in Y. The result is a single stable suggestion across all 8 sort directions.

1.5 — UI Integration

The suggestion is recomputed and pushed into the row-tolerance field automatically whenever the user changes:

  • Scope (Full Project ↔ Specific Floor)
  • Method (Smart Numbering ↔ Choose Your Way)
  • Sort direction (any of 8)
  • Reference floor (full project mode)
  • Specific floor (specific-floor mode)

A small purple hint label below the field explains the reasoning ("🪄 Auto-set to 131 cm — confident bimodal split; drift 87 cm, row separation 6.4 m"). The user can override the value at any time — auto-detection never locks the field.

2 · Zone-Aware Span Cap — Chain-Bridging Fix

The Bug

In the orthogonal portion of mixed-grid buildings, the adjacent-gap clustering would chain across the whole plan: every consecutive pair fell within tolerance, but the cumulative drift across 30+ columns merged what should have been three rows into one super-row. The user reported numbers jumping from 111 to 117 with no apparent reason.

The Fix — Zone-Aware Span Cap in ClusterByZoneAndAxis

A second condition is now applied only in orthogonal zones (where the local zone rotation is < 2°): the cluster's total axis span must remain ≤ 2 × tolerance. This breaks the chain at the second tolerance, well before it can swallow a true between-row gap.

In tilted zones the span cap is not applied — the local axis frame there is reconstructed from the detected rotation, and tiny residual rotation errors (1°) artificially inflate the local-axis span of a single physical line. Capping there would falsely split that line and re-interleave columns by ProjectX. Hence the predicate-driven design.

bool IsOrthogonalZone(ColumnGroup g) => Math.Abs(Rotation(g)) < 2.0;
double maxSpan = 2.0 * tolerance;

// During clustering:
bool spanOk = !currentCap || (a - firstAxis) <= maxSpan;
if (z == currentZone && gap <= tolerance && spanOk)
    current.Add(ordered[i]);

The 2° threshold sits below the zone classifier's 3° zone-separation threshold, so a tilted zone (≥3°) never qualifies. A "physically orthogonal" zone whose grids drift by 1°–2° still benefits from the cap.

✓ Shipped — 2 new unit tests cover both directions

3 · Manual Mark Correction

Motivation

No automatic algorithm can reach 100% on every plan. The engineer needs an escape hatch — a way to fix the last 1% by hand without re-running the whole numbering. This is the Manual Mark Correction command.

Workflow

  1. Engineer clicks the Edit Mark button on the ribbon.
  2. Revit prompts: "Pick a column".
  3. A small popup appears with the column's current mark, an editable field, Save & Continue, Save & Finish, and Cancel.
  4. On save: the new mark is written, and is automatically propagated up and down the stack to every other instance of the same multi-story column group (using the same XY-position equivalence the numbering engine uses).
  5. Save & Continue immediately re-prompts for the next column; Save & Finish closes.
  6. Esc at the picker closes cleanly.
✓ Confirmed by user in Revit
